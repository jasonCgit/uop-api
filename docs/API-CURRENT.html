Current API Reference All endpoints served by FastAPI backend — Base URL: http://localhost:8000

Related docs: Architecture Guide  |  API_DATA — detailed schemas & payloads

⚠
Mock Data: All endpoints currently return deterministic mock data from in-memory Python data structures. No external services are called. See the Mock Data section at the end for details.

Common Query Parameters

All dashboard endpoints accept these filter parameters. Filters use AND logic — all active filters are additive.

ParameterTypeFormatDescription
lobstring[]?lob=AWM&lob=CIBFilter by Line of Business
subLobstring[]?subLob=Asset+ManagementFilter by Sub-LOB
ctostring[]?cto=Jon+GlennieFilter by CTO name
cbtstring[]?cbt=Kalpesh+NarkhedeFilter by CBT name
sealstring[]?seal=16649&seal=35115Filter by application SEAL ID
statusstring[]?status=criticalFilter by computed health status
searchstring?search=MorganFree-text search on name or SEAL

Multi-value params use repeated keys: ?lob=AWM&lob=CIB (not comma-separated).

Application Endpoints

GET /api/applications/enriched

Full application catalog with computed health data. See Architecture Guide for the enrichment pipeline.

Available health indicator types for exclusion filtering.

["Process Group", "Service", "Synthetic"]

Exclude indicator types at the deployment level. Same request/response format.

PUT /api/applications/{app_id}/teams

Request: { "team_ids": [1, 5, 12] }

AURA AI Chat Endpoint

POST /api/aura/chat

Streams responses via Server-Sent Events (SSE).

Request

{
  "message": "What is the current health status of Morgan Money?",
  "conversation_id": "conv-123",
  "context": { "current_page": "dashboard", "active_filters": { "lob": ["AWM"] } }
}

SSE Event Protocol

EventPayloadDescription
meta{ message_id, timestamp }Sent first — conversation metadata
block{ type, data }Content block (text, metric_cards, status_list, chart, recommendations)
followups["question1", ...]Suggested follow-up questions
done{}Stream complete signal

Developer Guide: Replacing Mock Data

  1. Keep the same response schemas — the frontend depends on exact field names and shapes
  2. Add environment variables for external service URLs and credentials
  3. Replace data sources one at a time — start with APPS_REGISTRY (V12), then incidents (Unified Reporting), then graph (ERMA/V12)
  4. Cache expensive computations — maintain a caching strategy with TTL for live data
  5. Filter logic stays the same — works on any array of app objects matching the schema
  6. Test with mock data first — verify frontend behavior, then swap in live API calls
